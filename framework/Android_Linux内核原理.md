### 1.2 Android Linux 内核 vs 标准 Linux 内核

```
┌─────────────────────────────────────────────────────────────────────┐
│              Android Linux 内核 vs 标准 Linux 内核                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Android 并非直接使用原版 Linux 内核，而是做了大量定制：             │
│                                                                      │
│  ┌──────────────┬─────────────────────┬──────────────────────────┐  │
│  │ 维度          │ 标准 Linux 内核     │ Android Linux 内核       │  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ IPC 机制      │ 管道/消息队列/共享  │ Binder（一次拷贝，安全） │  │
│  │              │ 内存/Socket         │                          │  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ 共享内存      │ System V/POSIX shm  │ Ashmem（可回收、引用计数）│  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ 内存分配      │ CMA/DMA            │ ION → DMA-BUF heaps     │  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │

│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ 日志系统      │ dmesg/syslog       │ logd + /dev/log（分环形） │  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │

│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ 压缩交换      │ swap 分区/文件      │ zRAM 内存压缩           │  │
│  ├──────────────┼─────────────────────┼──────────────────────────┤  │
│  │ 文件系统      │ ext4/xfs/btrfs     │ ext4/f2fs/erofs          │  │
│  └──────────────┴─────────────────────┴──────────────────────────┘  │
│                                                                      │

│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │  GKI 架构（Android 12+）                                       │ │
│  │                                                                 │ │
│  │  ┌────────────────────────────────────────┐                    │ │
│  │  │         GKI 内核（Google 维护）         │                    │ │
│  │  │   通用代码 + Android 特有补丁           │                    │ │
│  │  └──────────────────┬─────────────────────┘                    │ │
│  │                     │ KMI (Kernel Module Interface)            │ │
│  │                     │ 稳定的模块接口                           │ │
│  │  ┌──────────────────┴─────────────────────┐                    │ │
│  │  │         厂商内核模块 (Vendor Modules)    │                    │ │
│  │  │   SoC 驱动 + 设备特定驱动               │                    │ │
│  │  └────────────────────────────────────────┘                    │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
```

## 2. 进程管理子系统

### 2.1 设计初衷与核心概念

```
┌─────────────────────────────────────────────────────────────────────┐
│                    进程管理：操作系统的核心                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  设计初衷：                                                         │
│  ──────────                                                         │
│  让多个程序"同时"运行在有限的 CPU 上，同时保证：                   │
│  • 隔离性：一个程序崩溃不影响其他程序                               │
│  • 公平性：每个程序都能获得合理的 CPU 时间                         │
│  • 安全性：程序不能随意访问其他程序的数据                           │
│                                                                      │
│  Linux 进程模型的核心设计哲学：                                     │
│  ─────────────────────────────                                      │
│                                                                      │
│  "Everything is a process (task)"                                   │
│                                                                      │
│  在 Linux 内核中，进程和线程并无本质区别，                          │
│  它们都是一个 task_struct 结构体的实例，                             │
│  区别仅在于是否共享资源（地址空间、文件描述符等）。                 │
│                                                                      │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │                    进程的生命周期状态                         │    │
│  │                                                              │    │
│  │           fork()              schedule()                     │    │
│  │   创建 ────────→ TASK_RUNNING ──────────→ CPU 执行          │    │
│  │                       ↑                      │               │    │
│  │                       │                      │               │    │
│  │              唤醒     │                      │ 等待 I/O      │    │
│  │          wake_up()    │                      │ 等待信号      │    │
│  │                       │                      │ 等待锁        │    │
│  │                       │                      ↓               │    │
│  │              TASK_INTERRUPTIBLE / TASK_UNINTERRUPTIBLE       │    │
│  │                       (睡眠/阻塞状态)                        │    │
│  │                                                              │    │
│  │          exit()                                              │    │
│  │   运行 ────────→ TASK_ZOMBIE ──────→ 父进程 wait() 回收     │    │
│  │                  (僵尸状态)                                   │    │
│  │                                                              │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  对 Android 的意义：                                                │
│  ────────────────                                                   │
│  每个 Android App 运行在独立的 Linux 进程中（由 Zygote fork），     │
│  拥有独立的 UID/GID，享受内核提供的进程隔离与资源管理。             │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 task_struct 数据结构

```
┌─────────────────────────────────────────────────────────────────────┐
│                    task_struct：进程描述符                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  task_struct 是 Linux 内核中最核心的数据结构之一，                   │
│  每个进程/线程对应一个 task_struct 实例，                            │
│  它包含了进程运行所需的全部信息。                                    │
│                                                                      │
│  // 源码路径: include/linux/sched.h                                  │
│  struct task_struct {                                                 │
│      // ① 调度信息                                                  │
│      volatile long state;           // 进程状态                     │
│      int prio, static_prio, normal_prio;  // 调度优先级             │
│      const struct sched_class *sched_class; // 调度类               │
│      struct sched_entity se;        // CFS 调度实体                 │
│                                                                      │
│      // ② 进程标识                                                  │
│      pid_t pid;                     // 进程 ID                      │
│      pid_t tgid;                    // 线程组 ID（= 主线程 pid）    │
│      struct task_struct *parent;    // 父进程                       │
│      struct list_head children;     // 子进程链表                   │
│                                                                      │
│      // ③ 内存管理                                                  │
│      struct mm_struct *mm;          // 内存描述符（虚拟地址空间）    │
│                                                                      │
│      // ④ 文件系统                                                  │
│      struct files_struct *files;    // 打开的文件描述符表           │
│      struct fs_struct *fs;          // 文件系统信息（当前目录等）    │
│                                                                      │
│      // ⑤ 信号处理                                                  │
│      struct signal_struct *signal;  // 信号处理信息                  │
│                                                                      │
│      // ⑥ 安全与权限                                                │
│      const struct cred *cred;       // 权限凭证（uid, gid）        │
│                                                                      │
│      // ⑦ 命名空间                                                  │
│      struct nsproxy *nsproxy;       // 命名空间（网络、PID等隔离）  │
│                                                                      │
│      // ... 数百个字段 ...                                          │
│  };                                                                  │
│                                                                      │
│  关键设计细节：                                                      │
│  ──────────────                                                      │
│                                                                      │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │         task_struct 在内核栈中的位置                           │  │
│  │                                                                │  │
│  │  高地址 ┌──────────────────────┐                              │  │
│  │         │    内核栈 (8KB/16KB) │                              │  │
│  │         │         ...          │  ← 内核栈从高向低增长       │  │
│  │         │         ...          │                              │  │
│  │         │         ↓            │                              │  │
│  │         │                      │                              │  │
│  │         │         ↑            │                              │  │
│  │         │    thread_info       │  ← 存放指向 task_struct 指针 │  │
│  │  低地址 └──────────────────────┘                              │  │
│  │                                                                │  │
│  │  通过 current 宏可快速获取当前进程的 task_struct：             │  │
│  │  #define current  get_current()                                │  │
│  │  // ARM64: 通过 sp_el0 寄存器直接获取                        │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                      │
│  pid vs tgid 的区别（理解 Linux 线程模型的关键）：                  │
│  ──────────────────────────────────────────────                      │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │  一个 App 进程 (pid=1000, tgid=1000)                       │     │
│  │                                                             │     │
│  │  ┌──────────────────┐  ┌───────────────┐  ┌────────────┐  │     │
│  │  │ 主线程            │  │ 工作线程 1     │  │ 工作线程 2 │  │     │
│  │  │ pid=1000          │  │ pid=1001      │  │ pid=1002   │  │     │
│  │  │ tgid=1000         │  │ tgid=1000     │  │ tgid=1000  │  │     │
│  │  └──────────────────┘  └───────────────┘  └────────────┘  │     │
│  │                                                             │     │
│  │  getpid() 系统调用返回的是 tgid（对用户态隐藏线程细节）    │     │
│  │  gettid() 系统调用返回的是真正的 pid（内核级线程 ID）      │     │
│  └────────────────────────────────────────────────────────────┘     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 进程创建：fork / clone / exec

```
┌─────────────────────────────────────────────────────────────────────┐
│                    进程创建三部曲                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Linux 创建新进程的三个核心系统调用：                                │
│                                                                      │
│  ┌─────────┬──────────────────────────────────────────────────────┐ │
│  │ 调用     │ 说明                                                │ │
│  ├─────────┼──────────────────────────────────────────────────────┤ │
│  │ fork()  │ 完全复制父进程，创建子进程                          │ │
│  │         │ 返回值：父进程得到子进程 PID，子进程得到 0          │ │
│  ├─────────┼──────────────────────────────────────────────────────┤ │
│  │ clone() │ 可选择性地共享资源（灵活版 fork）                   │ │
│  │         │ 通过 flags 控制共享哪些资源 → 用于创建线程         │ │
│  ├─────────┼──────────────────────────────────────────────────────┤ │
│  │ exec()  │ 替换当前进程的内存映像为新程序                      │ │
│  │         │ 通常在 fork() 之后调用                              │ │
│  └─────────┴──────────────────────────────────────────────────────┘ │
│                                                                      │
│  fork() 内部流程：                                                   │
│  ─────────────────                                                   │
│                                                                      │
│  父进程 ──fork()──→ do_fork() ──→ copy_process()                   │
│                                        │                             │
│                    ┌───────────────────┼──────────────────────┐      │
│                    ↓                   ↓                      ↓      │
│              dup_task_struct()   copy_mm()              copy_files() │
│              复制 task_struct    复制地址空间            复制文件表   │
│                    │                   │                      │      │
│                    │            ┌──────┴──────┐               │      │
│                    │            ↓             ↓               │      │
│                    │      Copy-On-Write   页表项复制          │      │
│                    │      (写时复制)      标记为只读          │      │
│                    │                                          │      │
│                    └──────────────────────────────────────────┘      │
│                                        │                             │
│                                        ↓                             │
│                                  wake_up_new_task()                  │
│                                  将子进程加入调度队列                │
│                                                                      │
│  Copy-On-Write（写时复制）——fork 的核心优化：                       │
│  ──────────────────────────────────────────                          │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  fork() 时并不真正复制物理内存页，而是让父子进程共享同一份     │ │
│  │  物理页，并将页表项标记为"只读"。                              │ │
│  │                                                                 │ │
│  │  fork() 之后：                                                  │ │
│  │  ┌──────────┐        ┌──────────────────┐                      │ │
│  │  │ 父进程    │──────→ │  物理页（只读）   │ ←────── ┌──────────┐│ │
│  │  │ 页表项    │        │  引用计数 = 2     │        │ 子进程   ││ │
│  │  │ (只读)    │        │                   │        │ 页表项   ││ │
│  │  └──────────┘        └──────────────────┘        │ (只读)   ││ │
│  │                                                    └──────────┘│ │
│  │  任一进程写入时触发 Page Fault：                                │ │
│  │  ┌──────────┐        ┌──────────────────┐  复制   ┌──────────┐│ │
│  │  │ 父进程    │──────→ │  原物理页         │ ←─────→│ 新物理页 ││ │
│  │  │ 页表项    │        │  引用计数 = 1     │        │ 子进程   ││ │
│  │  │ (可写)    │        │                   │        │ 专属     ││ │
│  │  └──────────┘        └──────────────────┘        └──────────┘│ │
│  │                                                                 │ │
│  │  好处：fork() + exec() 的常见模式中，子进程立即替换内存映像，  │ │
│  │        COW 使得 fork 几乎零开销。                               │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  Android 中的应用：                                                  │
│  ─────────────────                                                   │
│  Zygote fork App 进程时正是利用了 COW 机制：                        │
│  • Zygote 预加载的 Framework 类、资源等与 App 进程共享物理内存     │
│  • 只有 App 自身的代码和数据才会触发实际的内存分配                 │
│  • 大大减少了 App 启动时的内存开销和启动时间                       │
│                                                                      │
│  clone() flags 参数详解：                                            │
│  ────────────────────────                                            │
│                                                                      │
│  ┌────────────────┬──────────────────────────────────────────────┐  │
│  │ Flag            │ 效果                                        │  │
│  ├────────────────┼──────────────────────────────────────────────┤  │
│  │ CLONE_VM       │ 共享虚拟内存空间 → 创建线程的关键 flag      │  │
│  │ CLONE_FS       │ 共享文件系统信息（工作目录等）               │  │
│  │ CLONE_FILES    │ 共享文件描述符表                              │  │
│  │ CLONE_SIGHAND  │ 共享信号处理函数                              │  │
│  │ CLONE_THREAD   │ 放入同一线程组 (tgid 相同)                   │  │
│  │ CLONE_NEWPID   │ 新的 PID 命名空间（容器化）                 │  │
│  └────────────────┴──────────────────────────────────────────────┘  │
│                                                                      │
│  fork() = clone(最少共享)  →  创建独立进程                          │
│  pthread_create() = clone(CLONE_VM|CLONE_FS|CLONE_FILES|            │
│                          CLONE_SIGHAND|CLONE_THREAD)                │
│                        →  创建共享一切的线程                        │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.4 进程调度：CFS 完全公平调度器

```
┌─────────────────────────────────────────────────────────────────────┐
│                    CFS 完全公平调度器                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  设计初衷：                                                         │
│  ──────────                                                         │
│  传统 O(1) 调度器基于固定时间片，优先级高的进程获得更多时间片。     │
│  CFS（Completely Fair Scheduler）的哲学完全不同：                   │
│  "理想情况下，n 个进程应各占 1/n 的 CPU 时间"                      │
│                                                                      │
│  核心概念——虚拟运行时间 (vruntime)：                                │
│  ──────────────────────────────────                                  │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  vruntime = 实际运行时间 × (NICE_0_LOAD / 进程权重)            │ │
│  │                                                                 │ │
│  │  • 权重高的进程（高优先级）：vruntime 增长慢 → 更多 CPU 时间  │ │
│  │  • 权重低的进程（低优先级）：vruntime 增长快 → 更少 CPU 时间  │ │
│  │  • 调度器始终选择 vruntime 最小的进程运行                      │ │
│  │                                                                 │ │
│  │  示例：两个进程，权重比 2:1                                    │ │
│  │                                                                 │ │
│  │  时间 →    0    1    2    3    4    5    6                      │ │
│  │  进程 A:  [==] [==] [  ] [==] [==] [  ] [==]  运行 4/6        │ │
│  │  进程 B:  [  ] [  ] [==] [  ] [  ] [==] [  ]  运行 2/6        │ │
│  │                                                                 │ │
│  │  进程 A 的 vruntime 增长速度是 B 的一半，                      │ │
│  │  所以 A 的 vruntime 更小 → 被更频繁调度                       │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  红黑树组织就绪队列：                                                │
│  ────────────────────                                                │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  CFS 使用红黑树按 vruntime 排序所有就绪进程：                  │ │
│  │                                                                 │ │
│  │              ┌───────────┐                                     │ │
│  │              │ vruntime=8│                                     │ │
│  │              └─────┬─────┘                                     │ │
│  │              ┌─────┴─────┐                                     │ │
│  │         ┌────┴───┐  ┌────┴───┐                                 │ │
│  │         │ vr=5   │  │ vr=12  │                                 │ │
│  │         └───┬────┘  └───┬────┘                                 │ │
│  │         ┌───┴───┐   ┌───┴───┐                                  │ │
│  │     ┌───┴──┐ ┌──┴──┐ ┌──┴──┐                                  │ │
│  │     │ vr=3 │ │vr=6 │ │vr=10│                                  │ │
│  │     └──────┘ └─────┘ └─────┘                                  │ │
│  │     ↑                                                           │ │
│  │     最左节点 = vruntime 最小 = 下一个被调度的进程              │ │
│  │                                                                 │ │
│  │  • 插入/删除/查找：O(log n)                                    │ │
│  │  • 选择下一个进程：O(1)（缓存了最左节点 rb_leftmost）         │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  // 源码路径: kernel/sched/fair.c                                    │
│  static struct sched_entity *pick_next_entity(                       │
│      struct cfs_rq *cfs_rq) {                                        │
│      struct rb_node *left = rb_first_cached(&cfs_rq->tasks_timeline);│
│      struct sched_entity *se = rb_entry(left, struct sched_entity,   │
│                                         run_node);                   │
│      return se;                                                      │
│  }                                                                   │
│                                                                      │
│  调度类层级：                                                        │
│  ────────────                                                        │
│  ┌──────────────┬──────────┬──────────────────────────────────────┐ │
│  │ 调度类        │ 优先级    │ 用途                                │ │
│  ├──────────────┼──────────┼──────────────────────────────────────┤ │
│  │ stop_sched   │ 最高      │ 停机任务（CPU 热插拔、迁移）       │ │
│  │ dl_sched     │ 很高      │ 实时 deadline 调度                  │ │
│  │ rt_sched     │ 高        │ 实时进程（FIFO/RR）                 │ │
│  │ fair_sched   │ 普通      │ CFS 普通进程 ← 绝大多数进程       │ │
│  │ idle_sched   │ 最低      │ 空闲任务（无进程可调度时运行）     │ │
│  └──────────────┴──────────┴──────────────────────────────────────┘ │
│                                                                      │
│  调度顺序：stop → dl → rt → fair → idle                            │
│  只有高优先级调度类无就绪进程时，才轮到低优先级调度类。              │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.1 设计初衷：虚拟内存的价值

```
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │

│  │  ┌──────┬──────┬──────┬──────┬──────┬──────┐                   │ │
```

### 3.4 物理内存分配：Buddy System 与 Slab 分配器

```
┌─────────────────────────────────────────────────────────────────────┐
│                    物理内存分配器                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Linux 内核使用两层内存分配体系：                                    │
│  Buddy System 管理大块内存，Slab 管理小对象。                       │
│                                                                      │
│  ═══════════════════════════════════════════                          │
│  Buddy System（伙伴系统）                                           │
│  ═══════════════════════════════════════════                          │
│                                                                      │
│  设计初衷：高效分配连续的物理页框，同时减少外部碎片。               │
│                                                                      │
│  原理：将空闲内存按 2 的幂次分成不同阶 (order) 的块：              │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  order 0: ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ...    1 页 (4KB)         │ │
│  │  order 1: ┌────┐ ┌────┐ ┌────┐ ...          2 页 (8KB)         │ │
│  │  order 2: ┌────────┐ ┌────────┐ ...          4 页 (16KB)        │ │
│  │  order 3: ┌────────────────┐ ...              8 页 (32KB)        │ │
│  │  ...                                                            │ │
│  │  order 10: ┌─────────────────────────────┐   1024 页 (4MB)      │ │
│  │                                                                 │ │
│  │  分配 order=2 (4页) 的过程：                                    │ │
│  │  ────────────────────────────                                   │ │
│  │  1. 查找 order=2 的空闲链表                                    │ │
│  │  2. 如果为空，向上查找 order=3                                 │ │
│  │  3. 找到 order=3 的块，分裂为两个 order=2 的块                 │ │
│  │  4. 一个返回给调用者，另一个放入 order=2 空闲链表              │ │
│  │                                                                 │ │
│  │  释放时的合并：                                                 │ │
│  │  ──────────────                                                 │ │
│  │  释放 order=2 的块时，检查其"伙伴"是否也空闲：                │ │
│  │  ┌────────┐ ┌────────┐                                         │ │
│  │  │ 释放块 │ │ 伙伴块 │ ← 如果伙伴也空闲，合并为 order=3 块  │ │
│  │  └────────┘ └────────┘                                         │ │
│  │  继续向上检查，递归合并，减少碎片。                             │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  ═══════════════════════════════════════════                          │
│  Slab 分配器                                                        │
│  ═══════════════════════════════════════════                          │
│                                                                      │
│  设计初衷：Buddy System 最小分配单位是一页（4KB），                 │
│  但内核中大量对象远小于 4KB（如 task_struct ~6KB、inode ~1KB），     │
│  Slab 分配器解决小对象的高效分配和缓存问题。                        │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  Slab 分配器架构：                                              │ │
│  │                                                                 │ │
│  │  ┌─────────────────────────────────────────────────────────┐   │ │
│  │  │              kmem_cache (对象缓存)                       │   │ │
│  │  │                                                          │   │ │
│  │  │  名称: "task_struct"                                    │   │ │
│  │  │  对象大小: 6KB                                           │   │ │
│  │  │  每 slab 对象数: 5                                       │   │ │
│  │  │                                                          │   │ │
│  │  │  ┌───────────┐ ┌───────────┐ ┌───────────┐              │   │ │
│  │  │  │  slab (满) │ │ slab(部分)│ │ slab (空) │              │   │ │
│  │  │  │ ┌──┬──┬──┐│ │ ┌──┬──┬──┐│ │ ┌──┬──┬──┐│              │   │ │
│  │  │  │ │██│██│██││ │ │██│██│  ││ │ │  │  │  ││              │   │ │
│  │  │  │ ├──┼──┼──┤│ │ ├──┼──┼──┤│ │ ├──┼──┼──┤│              │   │ │
│  │  │  │ │██│██│  ││ │ │  │  │  ││ │ │  │  │  ││              │   │ │
│  │  │  │ └──┴──┴──┘│ │ └──┴──┴──┘│ │ └──┴──┴──┘│              │   │ │
│  │  │  └───────────┘ └───────────┘ └───────────┘              │   │ │
│  │  │  ██ = 已分配    空白 = 空闲                              │   │ │
│  │  └─────────────────────────────────────────────────────────┘   │ │
│  │                                                                 │ │
│  │  分配流程：优先从"部分使用"的 slab 中分配                     │ │
│  │  释放流程：对象归还到对应 slab，slab 全空时可归还给 Buddy     │ │
│  │                                                                 │ │
│  │  Linux 现代实现使用 SLUB 分配器（简化版，性能更好）            │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  // 源码路径: mm/slub.c                                              │
│  // 创建缓存                                                        │
│  struct kmem_cache *task_struct_cachep =                              │
│      kmem_cache_create("task_struct",                                │
│          sizeof(struct task_struct),  // 对象大小                    │
│          __alignof__(struct task_struct),  // 对齐要求               │
│          SLAB_PANIC | SLAB_ACCOUNT,  // 标志                        │
│          NULL);  // 构造函数                                        │
│                                                                      │
│  // 分配对象（O(1)，从 per-CPU 缓存直接获取）                      │
│  struct task_struct *tsk = kmem_cache_alloc(task_struct_cachep,       │
│                                             GFP_KERNEL);            │
│  // 释放对象                                                        │
│  kmem_cache_free(task_struct_cachep, tsk);                           │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.5 页面回收与 OOM

```
┌─────────────────────────────────────────────────────────────────────┐
│                    页面回收机制                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  当物理内存不足时，内核需要回收已分配的页面。                        │
│  回收策略的核心：优先回收不活跃的、可再次从磁盘读取的页面。         │
│                                                                      │
│  LRU 链表（Least Recently Used）：                                  │
│  ──────────────────────────────                                      │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  内核维护两组 LRU 链表：                                       │ │
│  │                                                                 │ │
│  │  ① 匿名页 (Anonymous LRU)                                     │ │
│  │     ┌──────────────────────────────────────────────┐           │ │
│  │     │  Active    ←→ [页A] [页B] [页C]              │           │ │
│  │     │  Inactive  ←→ [页D] [页E] [页F]              │           │ │
│  │     └──────────────────────────────────────────────┘           │ │
│  │     堆/栈数据，无文件后端，回收需要写入 swap                   │ │
│  │                                                                 │ │
│  │  ② 文件页 (File-backed LRU)                                   │ │
│  │     ┌──────────────────────────────────────────────┐           │ │
│  │     │  Active    ←→ [页G] [页H] [页I]              │           │ │
│  │     │  Inactive  ←→ [页J] [页K] [页L]              │           │ │
│  │     └──────────────────────────────────────────────┘           │ │
│  │     文件缓存/共享库映射，回收后可从文件重新读取                │ │
│  │                                                                 │ │
│  │  回收优先级：                                                   │ │
│  │  Inactive File > Inactive Anon > Active File > Active Anon     │ │
│  │  （文件页优先回收，因为可以从文件重新加载，代价小）             │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  kswapd 后台回收：                                                   │
│  ─────────────────                                                   │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  空闲内存                                                       │ │
│  │   ↑                                                             │ │
│  │   │  ┌─────────────────┐                                       │ │
│  │   │  │  high watermark │  ← 空闲页充足，kswapd 停止回收       │ │
│  │   │  ├─────────────────┤                                       │ │
│  │   │  │  low watermark  │  ← kswapd 被唤醒，后台异步回收       │ │
│  │   │  ├─────────────────┤                                       │ │
│  │   │  │  min watermark  │  ← 直接回收（同步，阻塞分配者）     │ │
│  │   │  ├─────────────────┤                                       │ │
│  │   │  │       0         │  ← 触发 OOM Killer，杀进程释放内存  │ │
│  │   │  └─────────────────┘                                       │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 6. Binder IPC 机制

### 6.1 设计初衷：为什么不用传统 IPC

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Binder 的设计初衷                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Android 中每个 App 运行在独立的进程中，进程间通信 (IPC) 是基础需求。│
│  为什么不直接使用 Linux 原有的 IPC 机制？                           │
│                                                                      │
│  ┌──────────────┬──────────┬──────────┬──────────┬────────────────┐ │
│  │ IPC 机制      │ 拷贝次数 │ 安全验证 │ C/S 模型│ Android 适用性│ │
│  ├──────────────┼──────────┼──────────┼──────────┼────────────────┤ │
│  │ 管道 (pipe)   │ 2 次     │ 无       │ 不支持   │ 低效、功能弱  │ │
│  │ 消息队列      │ 2 次     │ 无       │ 不支持   │ 不安全        │ │
│  │ 共享内存      │ 0 次     │ 无       │ 不支持   │ 需额外同步    │ │
│  │ Socket        │ 2 次     │ 可扩展   │ 支持     │ 开销大        │ │
│  │ Binder        │ 1 次     │ 内核级   │ 天然支持 │ ★ 最佳选择   │ │
│  └──────────────┴──────────┴──────────┴──────────┴────────────────┘ │
│                                                                      │
│  Binder 的三大优势：                                                │
│  ─────────────────                                                   │
│                                                                      │
│  ① 性能：只需一次数据拷贝（vs 管道/消息队列/Socket 的两次）       │
│     发送方用户空间 → 接收方内核映射区（mmap）→ 接收方直接读取     │
│                                                                      │
│  ② 安全：内核驱动自动附加调用者的 UID/PID，不可伪造               │
│     传统 IPC 中发送方可以伪造身份信息，Binder 从内核层面保证真实性 │
│                                                                      │
│  ③ 架构：天然的 C/S（客户端/服务端）架构                           │
│     与 Android 的 Service 设计完美匹配（startService/bindService） │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

```
│  │  发送方用户空间 ──copy_from_user──→ 内核缓冲区                 │ │

│  │                                                                 │ │

│  │                                                                 │ │

│  Binder 只需一次拷贝：                                              │
│  ────────────────────                                                │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │

│  │                                                                 │ │
│  │  接收方进程调用 mmap() 时，Binder 驱动做了两件事：             │ │

│  │  ┌─────────────────┐                                           │ │
│  │  │ 发送方用户空间   │                                           │ │
│  │  │  ┌─────────────┐│                                           │ │
│  │  │  │  发送数据    ││                                           │ │
│  │  │  └──────┬──────┘│                                           │ │
│  │  └─────────│───────┘                                           │ │
│  │            │ copy_from_user (仅此一次拷贝)                      │ │
│  │            ↓                                                    │ │
│  │  ┌─────────────────────────────────────────────────┐           │ │
│  │  │              物理内存页                          │           │ │
│  │  │         (同时被两个虚拟地址映射)                 │           │ │
│  │  └───────────┬─────────────────────┬───────────────┘           │ │
│  │              ↑                     ↑                            │ │
│  │              │ 内核虚拟地址映射     │ 用户虚拟地址映射          │ │
│  │              │                     │                            │ │
│  │  ┌───────────┴──────┐  ┌──────────┴────────┐                  │ │
│  │  │ 内核空间          │  │ 接收方用户空间     │                  │ │

│  │  │  写入数据到此处)  │  │                    │                  │ │
│  │  └──────────────────┘  │  接收方直接读取，  │                  │ │
│  │                         │  无需 copy_to_user │                  │ │
│  │                         └───────────────────┘                  │ │
│  │                                                                 │ │
│  │  数据流向：                                                     │ │
│  │  发送方用户空间 → (copy_from_user) → 内核/接收方共享的物理页   │ │
│  │  → 接收方通过 mmap 直接读取                                    │ │
│  │                                                                 │ │
│  │  总计：仅 1 次数据拷贝                                         │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  // 源码路径: drivers/android/binder_alloc.c                         │
│  static int binder_alloc_mmap_handler(                               │
│      struct binder_alloc *alloc,                                     │
│      struct vm_area_struct *vma) {                                   │
│                                                                      │
│      // 用户空间虚拟地址                                            │
│      alloc->buffer = (void __user *)vma->vm_start;                   │
│                                                                      │
│      // 分配物理页并同时映射到用户空间和内核空间                    │
│      alloc->pages[i].page_ptr = alloc_page(GFP_KERNEL);             │
│      // 映射到用户空间 vma                                          │
│      vm_insert_page(vma, user_addr, page);                           │
│      // 映射到内核空间 (通过 kmap 或直接映射)                      │
│  }                                                                   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.2 内核同步原语

```
┌─────────────────────────────────────────────────────────────────────┐
│                    内核同步原语                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  多核 CPU 环境下，多个执行路径（进程/中断）可能同时访问共享数据，   │
│  内核提供了多种同步机制，适用于不同场景：                           │
│                                                                      │
│  ┌──────────────┬────────────┬──────────┬─────────────────────────┐ │

│  │ spinlock     │ 不可睡眠   │ 任意     │ 短临界区（几行代码）    │ │
│  │ (自旋锁)     │ 忙等待     │          │ 中断上下文、调度器内部 │ │
│  ├──────────────┼────────────┼──────────┼─────────────────────────┤ │
│  │ mutex        │ 可以睡眠   │ 进程上下文│ 长临界区（可能阻塞）   │ │
│  │ (互斥锁)     │ 让出 CPU   │          │ 文件操作、设备驱动     │ │
│  ├──────────────┼────────────┼──────────┼─────────────────────────┤ │
│  │ semaphore    │ 可以睡眠   │ 进程上下文│ 限制并发数（计数信号量）│ │
│  │ (信号量)     │            │          │ 有限资源池             │ │
│  ├──────────────┼────────────┼──────────┼─────────────────────────┤ │
│  │ rwlock       │ 不可睡眠   │ 任意     │ 读多写少的短临界区     │ │
│  │ (读写锁)     │            │          │ 路由表查找             │ │
│  ├──────────────┼────────────┼──────────┼─────────────────────────┤ │
│  │ RCU          │ 读不加锁   │ 任意     │ 读远多于写的场景       │ │
│  │              │ 写延迟释放 │          │ 网络路由表、进程列表   │ │
│  ├──────────────┼────────────┼──────────┼─────────────────────────┤ │
│  │ atomic       │ 无锁       │ 任意     │ 简单计数器             │ │
│  │ (原子操作)   │ CPU 指令级 │          │ 引用计数、统计计数     │ │
│  └──────────────┴────────────┴──────────┴─────────────────────────┘ │
│                                                                      │
│  RCU（Read-Copy-Update）——Linux 内核最优雅的同步机制：              │
│  ──────────────────────────────────────────────────                   │
│                                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                                                                 │ │
│  │  核心思想：读操作无锁，写操作先复制再修改，等所有读者退出后    │ │
│  │  才释放旧数据。                                                 │ │
│  │                                                                 │ │
│  │  读者 A ──────────[读旧数据]──────────                          │ │
│  │  读者 B ──────[读旧数据]──────────────                          │ │
│  │                                                                 │ │
│  │  写者: ① 复制旧数据                                            │ │
│  │        ② 修改副本                                               │ │
│  │        ③ 用新指针替换旧指针 (rcu_assign_pointer)                │ │
│  │        ④ 等待所有读者完成 (synchronize_rcu / call_rcu)          │ │
│  │        ⑤ 释放旧数据                                            │ │
│  │                                                                 │ │
│  │  读者 C ──────────────[读新数据]──────                          │ │
│  │                                                                 │ │
│  │  优势：读操作几乎零开销（仅 rcu_read_lock 禁止抢占）          │ │
│  │  代价：写操作需要额外的内存和延迟释放                          │ │
│  │                                                                 │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 10.1 SELinux 强制访问控制

```
┌─────────────────────────────────────────────────────────────────────┐
│                    SELinux 强制访问控制                               │
├─────────────────────────────────────────────────────────────────────┤

│  │                                                                 │ │

│  │  进程 A (uid=1000, context=untrusted_app)                       │ │

│  │  ① DAC 检查: uid 匹配 → 通过 ✓                                │ │

│  │     访问 app_data_file？ → 查询策略数据库                      │ │

│                                                                      │
│  SELinux 策略示例（Android .te 文件）：                              │
│  ──────────────────────────────────────                              │
│                                                                      │
│  # 允许 untrusted_app 域的进程                                      │
│  # 对 app_data_file 类型的文件                                      │
│  # 执行 read, write, open 操作                                      │
│  allow untrusted_app app_data_file:file { read write open };         │
│                                                                      │
│  # 禁止任何 App 直接访问 /dev/kmem                                 │

│  ├──────────────────┼──────────────────────────────────────────┤    │

│  │ untrusted_app    │ 第三方 App                                │    │
│  │ isolated_app     │ 隔离的 Service 进程                       │    │

│  └──────────────────┴──────────────────────────────────────────┘    │
│                                                                      │
```
