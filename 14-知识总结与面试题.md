## 2. 核心知识点速查

| 模块 | 核心知识点（3 句话） |
|------|----------------------|
| **Binder** | Binder 是 Android 自研的进程间通信机制，基于 mmap + 一次数据拷贝，相比 Socket 更高效、安全。Binder Driver 在 Kernel 中实现，Java 层通过 AIDL 生成 Stub/Proxy 代理类。ServiceManager 是 Binder 的命名服务，用于查找系统服务。 |
| **AMS/WMS** | AMS 管理 Activity 生命周期、Task、进程调度，负责 Activity 启动、切换、销毁。WMS 管理窗口的添加、布局、层级（Z-order）、输入焦点分配。二者协作：AMS 决定「启动什么」，WMS 决定「窗口如何显示、输入给谁」。 |
| **View 体系** | View 的 measure/layout/draw 由 ViewRootImpl.performTraversals() 驱动，遵循自上而下 measure、自上而下 layout、自上而下 draw。硬件加速下 draw 阶段只录制 DisplayList，不实际绘制。invalidate 仅触发 draw，requestLayout 触发 measure+layout+draw。 |
| **HWUI** | HWUI 是 Android 硬件加速渲染库，主线程录制 DisplayList，RenderThread 回放并生成 GPU 命令。RenderNode 存储一个 View 的 DisplayList 和属性，Canvas.drawXxx 会录制成 Op 指令。Skia 作为 2D 图形库负责路径、位图等的 GPU 加速绘制。 |
| **SurfaceFlinger** | SurfaceFlinger 是合成的核心服务，从各 App 的 BufferQueue 获取 Buffer，按 Z-order 合成到一块 framebuffer。HWC 合成由硬件完成（overlay），GPU 合成用于复杂场景。支持多屏、旋转、分屏等。 |
| **VSync** | VSync 是垂直同步信号，由 Display 硬件产生，用于对齐渲染节奏。VSync-app 驱动 App 的 Choreographer.doFrame，VSync-sf 驱动 SurfaceFlinger 合成。VSYNC-app 和 VSYNC-sf 有 offset 错开，确保 App 在 SF 合成前完成渲染。 |
| **SystemUI** | SystemUI 是系统状态栏、导航栏、通知栏等 UI 的宿主进程。状态栏通过 WindowManager.addView 以 TYPE_STATUS_BAR 类型添加。使用 Dagger 做依赖注入，模块化程度高。 |
| **Launcher** | Launcher3 是 AOSP 默认桌面，桌面图标和 Widget 在 CellLayout 网格中排列。CellLayout 根据 spanX/spanY 计算每个 item 的格子位置。支持拖拽、文件夹、工作区切换等。 |

## 3. 20 道高频面试题与参考答案

### Q1. 描述 Activity 启动的完整流程（从 Launcher 点击到首帧上屏）

**答：** 用户点击图标 → Launcher 通过 Binder 调用 `ActivityTaskManagerService.startActivity()` → AMS 做权限校验、创建 ActivityRecord、TaskRecord → 若目标进程不存在则 fork Zygote 子进程启动 App → 目标进程 `ActivityThread.main()` 启动，绑定 Application，创建 Context → `handleLaunchActivity()` 创建 Activity、调用 `onCreate` → WMS 通过 `addWindow` 添加 DecorView → `ViewRootImpl.performTraversals()` 触发 measure/layout/draw → HWUI 渲染到 Surface，queueBuffer 到 BufferQueue → SurfaceFlinger 在 VSync-sf 时合成 → 首帧上屏。核心路径：Launcher → AMS → Zygote → ActivityThread → WMS → ViewRootImpl → HWUI → SurfaceFlinger。

---

### Q2. Binder 通信的原理，为什么 Android 选择 Binder 而不是 Socket？

**答：** Binder 原理：客户端持有服务端 Binder 的引用，调用时通过 Binder Driver 将数据拷贝到内核，再拷贝到服务端进程（一次拷贝 + mmap 共享）。选择 Binder 的原因：① 性能更好，Socket 需要两次拷贝，Binder 仅一次；② 安全性，Binder 可验证 UID/PID，Socket 需应用层实现；③ 面向对象，Binder 天然支持引用计数和对象传递；④ 与 Android 的 C/S 架构契合，系统服务多以 Binder 暴露。

---

### Q3. ViewRootImpl.performTraversals() 做了什么？三个阶段各自的作用？

**答：** `performTraversals()` 是 View 树 Traversal 的入口，一次调用完成 measure、layout、draw 三阶段。① **measure 阶段**：`performMeasure()` 自顶向下递归，每个 View 的 `onMeasure` 确定自己的宽高，通过 `MeasureSpec` 传递约束；② **layout 阶段**：`performLayout()` 自顶向下递归，每个 View 的 `onLayout` 确定子 View 的位置；③ **draw 阶段**：`performDraw()` 调用 `ThreadedRenderer.draw()`，主线程仅录制 DisplayList，不实际绘制。硬件加速下 draw 是「录制」而非「渲染」。

---

### Q4. invalidate() 和 requestLayout() 的区别？各自触发什么流程？

**答：** **invalidate()**：标记当前 View 需要重绘，只触发 **draw** 阶段。内部会向父 View 传播 dirty 区域，最终在下一帧的 `performTraversals` 中执行 `performDraw()`。不触发 measure 和 layout。适用于仅视觉变化（如颜色、背景）的场景。

**requestLayout()**：标记当前 View 及其父链需要重新测量和布局，触发 **measure + layout + draw** 三阶段。会向上传播到 ViewRootImpl，在下一帧的 `performTraversals` 中完整执行。适用于尺寸或位置变化的场景。

---

### Q5. 硬件加速下 Canvas.drawXxx() 发生了什么？与软件渲染有何不同？

**答：** **硬件加速下**：`Canvas.drawXxx()` 不会立即绘制，而是将绘制命令录制为 DisplayList 中的 **Op 指令**（如 DrawRectOp、DrawPathOp）。主线程完成后，RenderThread 在 GPU 上回放这些 Op，由 Skia 生成 OpenGL/Vulkan 命令。优势：主线程只做轻量录制，实际绘制在独立线程，可利用 GPU 并行。

**软件渲染**：Canvas 对应 `SkBitmap`，`drawXxx` 直接在 CPU 上操作位图像素，每帧将整块 bitmap 上传到 GPU 或通过 Skia 软件光栅化。主线程负担重，无法利用 GPU 加速。

---

### Q6. 什么是 DisplayList？RenderNode 的作用是什么？

**答：** **DisplayList**：一系列绘制操作（Op）的指令序列，如 DrawRectOp、DrawPathOp、DrawDisplayListOp 等。主线程在 draw 阶段通过 RecordingCanvas 录制，RenderThread 按序回放执行。DisplayList 可缓存复用，若 View 未变化则不需重新录制。

**RenderNode**：每个 View 对应一个 RenderNode，用于存储该 View 的 DisplayList 和属性（transform、alpha、clip 等）。RenderNode 构成树形结构，子节点的 DisplayList 作为父节点的 `DrawDisplayListOp` 嵌入。属性变化时只需更新 RenderNode 的属性，不必重录 DisplayList。

---

### Q7. 主线程和 RenderThread 如何协作？syncFrameState 做了什么？

**答：** 主线程负责 measure/layout/draw 中的 DisplayList 录制，RenderThread 负责 GPU 渲染和 queueBuffer。**syncFrameState** 是二者同步的桥梁：主线程在 `performDraw` 末尾调用，将当前帧的 DisplayList、属性、动画状态等打包，通过 `DrawFrameTask` 交给 RenderThread。RenderThread 收到后：更新 RenderNode 树、回放 DisplayList、执行 GPU 绘制、eglSwapBuffers、queueBuffer。保证了主线程「录制」和 RenderThread「渲染」的时序一致。

---

### Q8. Surface、SurfaceView、TextureView 的区别？各自的 BufferQueue 模型？

**答：** **Surface**：绘图表面，持有 BufferQueue 的 Producer 端，App 通过 Surface 获得 GraphicBuffer 并绘制，绘制完成后 queueBuffer。

**SurfaceView**：有独立 Surface，**独立于 App 的 View 层级**，其 Surface 由 WMS 在 SurfaceFlinger 侧创建，有独立的 BufferQueue，可直接合成到屏幕，不经过 App 的 DecorView。适合视频、相机等需要低延迟的场景，但无法做透明度、变换等。

**TextureView**：没有独立 Surface，**作为普通 View 嵌入 View 树**，内容通过 SurfaceTexture 渲染到 App 的 Surface 上，再随 App 的 Buffer 一起合成。支持 transform、alpha 等，但多一层合成，延迟略高。

---

### Q9. SurfaceFlinger 的合成流程是什么？HWC 和 GPU 合成的区别？

**答：** 合成流程：① VSync-sf 触发 `onMessageRefresh`；② `acquireBuffer` 从各 Layer 的 BufferQueue 获取 Buffer；③ `rebuildLayerStacks` 计算可见区域、Z-order；④ `prepareFrame` 向 HWC 询问合成策略；⑤ `doComposition` 执行合成；⑥ 提交到 Display。

**HWC 合成**：由 Display 硬件的 Overlay 引擎完成，直接将 Layer 的 Buffer 叠加显示，不占用 GPU，功耗低，适合简单场景。

**GPU 合成**：由 GPU 将多个 Layer 绘制到一块 framebuffer，再交给 Display。用于复杂效果（如缩放、旋转、混合），占用 GPU 算力。

---

### Q10. BufferQueue 的生产者-消费者模型是如何工作的？三重缓冲的优势？

**答：** **生产者**（App/HWUI）通过 `dequeueBuffer` 获取空闲 Buffer，绘制完成后 `queueBuffer` 放回；**消费者**（SurfaceFlinger）通过 `acquireBuffer` 获取已绘制的 Buffer，合成显示后 `releaseBuffer` 归还。Buffer 在三种状态间流转：Free → Dequeued → Queued → Acquired → Free。

**三重缓冲**：除了当前显示的 Buffer（front）和 App 正在绘制的 Buffer（back），再多一块 Buffer 用于预渲染。优势：App 可以在上一帧还在显示时就开始绘制下一帧，减少因等待 acquireBuffer 导致的卡顿；当 App 偶发掉帧时，SF 仍有可用 Buffer，避免画面撕裂或重复显示旧帧。

---

### Q11. VSync 信号是如何从硬件传递到 App 的 Choreographer 的？

**答：** Display 硬件在每帧扫描结束时产生 VSync 信号 → 由 Display HAL 上报 → SurfaceFlinger 的 EventThread 接收并产生 `DisplayEventReceiver` 事件 → 通过 `DispSync` 预测后续 VSync 时间点 → App 的 Choreographer 通过 `FrameDisplayEventReceiver`（继承 DisplayEventReceiver）向 SurfaceFlinger 的 EventThread 注册监听 → EventThread 通过 Binder/共享内存将 VSync 时间戳回调给 Choreographer → Choreographer 在收到 VSync 后调用 `doFrame` 执行一帧的 input、animation、traversal。

---

### Q12. VSYNC-app 和 VSYNC-sf 的 offset 机制，为什么需要错开？

**答：** VSYNC-app 和 VSYNC-sf 是同一物理 VSync 信号的两个逻辑相位，通过 **Phase offset** 错开触发时间。例如：VSync 0ms 时发 VSYNC-app，4ms 时发 VSYNC-sf。**原因**：App 的 measure/layout/draw 和 queueBuffer 需要时间，若 VSYNC-app 和 VSYNC-sf 同时触发，SF 开始合成时 App 可能还没完成渲染，导致 SF 只能使用上一帧的旧 Buffer，造成掉帧。错开后，App 在 VSYNC-app 时开始绘制，有约 4–8ms 窗口在 VSYNC-sf 前完成 queueBuffer，SF 能拿到最新帧，保证流水线顺畅。

---

### Q13. 什么是掉帧（Jank）？从渲染链路角度分析掉帧的可能原因？

**答：** **掉帧**：一帧的渲染时间超过 16.6ms（60fps），导致该帧无法在预期 VSync 前完成，用户感知为卡顿。

**可能原因**：① **主线程繁忙**：measure/layout/draw 超时，如复杂布局、主线程 GC、锁竞争；② **RenderThread 繁忙**：DisplayList 过大、复杂 path、过度 overdraw；③ **SurfaceFlinger 慢**：Layer 过多、GPU 合成压力大；④ **Binder 阻塞**：主线程等待 Binder 返回；⑤ **VSync 迟到**：调度延迟、CPU 频率不足；⑥ **Buffer 不足**：BufferQueue 饿死，等待 dequeueBuffer。

---

### Q14. 如何使用 Perfetto 分析一个掉帧问题？你的分析步骤是什么？

**答：** ① 用 `perfetto -o trace.pb -t 5s sched gfx view am wm` 抓 Trace；② 在 Perfetto UI 打开 trace，找到目标进程的 `Choreographer#doFrame` slice；③ 查看 doFrame 是否超过 16ms，若超则定位是 INPUT/ANIMATION/TRAVERSAL 中哪一段耗时；④ 展开 TRAVERSAL，看 `performTraversals`、`measure`、`layout`、`draw` 的耗时分布；⑤ 若 draw 慢，查看 RenderThread 的 `DrawFrame`、`flushCommands` 等；⑥ 用 `frame_timeline` 表查 `frame_state=0`（Jank）的帧，分析其 delay_reason、drop_reason；⑦ 结合 sched 看是否有主线程被抢占、CPU 频率不足等问题。

---

### Q15. SystemUI 的状态栏是如何添加到屏幕上的？窗口类型是什么？

**答：** SystemUI 进程启动时，`StatusBar` 组件通过 `WindowManager.addView()` 添加状态栏 View。窗口类型为 `TYPE_STATUS_BAR`（或 `TYPE_APPLICATION_OVERLAY` 等，视版本而定），该类型具有最高层级，会覆盖在普通 App 之上。添加时指定 `LayoutParams` 的宽高、gravity（通常 TOP）、flags（如 `FLAG_NOT_FOCUSABLE`）。WMS 根据窗口类型分配 Z-order，状态栏位于 system overlay 层，由 SurfaceFlinger 按 Z-order 合成到屏幕顶部。

---

### Q16. SystemUI 中 Dagger 依赖注入是如何使用的？

**答：** SystemUI 使用 Dagger 2 做依赖注入。核心用法：① 用 `@Inject` 标记构造函数或字段，Dagger 自动生成依赖；② 用 `@Module` 定义提供依赖的模块，`@Provides` 方法返回具体实现；③ 用 `@Component` 定义注入入口，关联 Module 和待注入的类；④ `@Singleton`、`@ActivityScope` 等控制生命周期。例如 `StatusBar` 依赖 `CommandQueue`、`NotificationShadeWindowController` 等，通过 `StatusBarComponent` 注入，避免手动 new 和传递，便于单测和模块替换。

---

### Q17. Launcher3 的桌面布局是如何管理的？CellLayout 的网格计算？

**答：** Launcher3 桌面由 `Workspace` 管理多个 `CellLayout` 页面，每页是一个网格。**CellLayout** 继承自 `ViewGroup`，网格由 `mCountX`、`mCountY` 定义（如 4×5），每个格子大小由 `mCellWidth`、`mCellHeight` 计算。拖拽的 item 有 `spanX`、`spanY` 占据多格。布局时，`CellLayout` 根据 item 的 `cellX`、`cellY`、`spanX`、`spanY` 计算其 left/top/right/bottom，考虑边距、padding 后调用 `layout()`。使用 `ItemConfiguration` 做拖放时的占位预布局。网格参数可从 `GridOption` 或配置读取。

---

### Q18. 应用冷启动优化你会从哪些方面入手？

**答：** ① **Application 与首屏 Activity**：减少 Application.onCreate 中的初始化，懒加载非关键逻辑；② **类加载**：避免首屏加载过多类，用 ClassLoader 预加载或异步加载；③ **布局**：减少层级、用 ViewStub/merge、异步 Inflation；④ **主线程**：避免主线程 IO、反序列化，用 IdleHandler 延后非关键任务；⑤ **GC**：减少启动路径上的内存分配，避免触发 GC；⑥ **多进程**：将 Heavy 组件放子进程，降低主进程启动负担；⑦ **Profile**：用 Perfetto 或 Systrace 抓启动 trace，定位 doFrame、class init、GC 等热点，逐项优化。

---

### Q19. ANR 的原因有哪些？如何通过 traces.txt 和 Perfetto 分析？

**答：** **ANR 原因**：① 主线程被阻塞（锁、Binder 同步调用、IO）；② BroadcastReceiver 超时（10s）；③ Service 超时（20s）；④ Input 无响应（5s）。

**traces.txt 分析**：`adb pull /data/anr/traces.txt`，查看主线程堆栈，找到 `BLOCKED`、`WAITING` 状态，看等待的锁或 Binder 调用来自哪个线程。

**Perfetto 分析**：抓 trace 含 `sched`、`binder`、`am`，找到 ANR 时间点，看主线程在做什么 slice；若有 `binder` 信息，查主线程是否在等 Binder 返回；用 `thread_state` 看是否 blocked。结合 `InputMethodManagerService`、`ActivityManager` 等系统服务的调用，定位阻塞源头。

---

### Q20. Android 的低内存管理机制是什么？ADJ 值的含义？

**答：** Android 通过 **LMK（Low Memory Killer）** 在内存不足时按 ADJ 从高到低杀进程回收内存。**ADJ (oom_adj)**：表示进程的「重要性」，值越大越容易被杀。常见 ADJ：`FOREGROUND_APP=0`（前台）、`VISIBLE_APP=1`（可见）、`PERCEPTIBLE_APP=2`（可感知，如后台音乐）、`BACKUP=3`、`CACHED_APP=15`（缓存）。系统根据进程状态（前台、可见、服务、缓存等）动态计算 ADJ。`dumpsys meminfo`、`procrank` 可查看各进程 ADJ。优化思路：前台进程 ADJ 最低，尽量保持可见状态，避免无谓后台常驻。

## 4. 面试技巧

### 4.1 回答框架

- **先说结论**：用一两句话概括答案，展示你理解核心概念。
- **再说流程**：按阶段、按步骤说明，体现系统性思维。
- **最后说源码/数据**：能点出关键类、方法、路径或数据指标，体现深度。

### 4.2 可以画图辅助

- 面试时可用白板画流程图、架构图。
- 例如：Activity 启动流程、Binder 通信模型、渲染三阶段、BufferQueue 状态机、VSync 相位等。
- 图能快速传递信息，节省口述时间。

### 4.3 主动提及 Perfetto 分析经验

- 面试官常问「如何分析卡顿」「如何定位 ANR」。
- 主动说「我用 Perfetto 抓 trace，从 frame_timeline 看 jank 帧，结合 doFrame 和 RenderThread 分析」。
- 展示你会用工具、有实战经验，是明显加分项。

### 4.4 从 App 开发经验过渡到 Framework 理解的话术

- 「我在做 App 时遇到过 X 问题，后来学了 Framework 才知道底层是 Y 机制」。
- 「以前只知道 invalidate 会刷新，学了 HWUI 后理解了 DisplayList 和 RenderThread 的协作」。
- 「5 年 App 经验让我对 measure/layout 有直觉，学 Framework 补齐了从 Choreographer 到 SurfaceFlinger 的全链路」。
- 这种过渡能体现：你有实践基础，且愿意深入底层，学习动机明确。
